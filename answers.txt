TASK 3 - SQL Injection Attacks


How I did it:
I created vulnerable.js with two functions: searchProduct() and login(). Both functions are VULNERABLE 
because they use template literals (backticks with ${variable}) to build SQL queries. This means user 
input gets inserted directly into the SQL string.

Then I ran 4 attacks by calling these functions with malicious payloads.

Attack 1 - Dump all products:
I called searchProduct("' OR 1=1--")
What happened: The OR 1=1 made the WHERE clause always true, so every product came back.
The actual query built was: SELECT * FROM products WHERE name LIKE '%' OR 1=1--%'
Result: All 5 products returned (attacker sees all cost prices)

Attack 2 - Login bypass:
I called login("admin'--", "anything")
What happened: The -- commented out the password check, so only username mattered.
The actual query: SELECT * FROM users WHERE username='admin'--' AND password='anything'
Result: Admin user returned without needing password

Attack 3 - Always true login:
I called login("' OR '1'='1", "' OR '1'='1")
What happened: OR '1'='1' is always true, so the WHERE clause matches every row.
Result: First user in database (admin) returned

Attack 4 - UNION attack to steal passwords:
I called searchProduct("' UNION SELECT id, username, password, role FROM users--")
What happened: UNION combined product search with users table, exposing all credentials.
Result: All 5 products PLUS all 3 users with their passwords came back

Commands I ran:
node vulnerable.js

TASK 4 - Fixing with Parameterised Queries


How I did it:
I created secure.js and rewrote the same functions but this time using ? placeholders instead of template literals.

For searchProductSafe(), I wrote:
const query = `SELECT * FROM products WHERE name LIKE '%' || ? || '%'`
const rows = db.prepare(query).all(name)

For loginSafe(), I wrote:
const query = `SELECT * FROM users WHERE username = ? AND password = ?`
const row = db.prepare(query).get(username, password)

The ? placeholders tell the database "put the user input here as DATA, not as SQL code".

When I tested the same attacks:
- searchProductSafe("' OR 1=1--") returned [] because it searched for that literal string
- searchProductSafe("' UNION SELECT...") returned [] because UNION is now just text to search for
- loginSafe("admin'--", "anything") returned null because username "admin'--" doesn't exist
- loginSafe("' OR '1'='1", "' OR '1'='1") returned null because those literal strings aren't in the database

Commands I ran:
node secure.js

Why parameterised queries prevent injection (in my own words):
When I use template literals like `WHERE name LIKE '%${name}%'`, the user's input becomes 
part of the SQL command. If they type a quote or semicolon, the database thinks that's 
part of the SQL syntax. With parameterised queries using ? placeholders, the database 
compiles the SQL structure FIRST. Then it knows exactly where the user data goes, and 
it treats whatever the user types as plain data - never as executable code. Even if 
someone types "' OR 1=1--", the database just sees it as a weird string to search for, 
not as SQL commands. The quotes and dashes lose their special meaning.

TASK 5 - Adding Input Validation


How I did it:
I updated secure.js to add validation functions that check input BEFORE it even reaches the database.

I created these validation rules:
- validateName(): checks string, at least 2 characters, no < > or ; symbols
- validateUsername(): checks string, no spaces, not empty
- validatePassword(): checks string, at least 6 characters

In each function, I check validation first. If validation fails, I print an error and return [] or null without touching the database.

Test results I got:

1. searchProductSafe('cement') -> PASSED validation, returned Cement product
2. searchProductSafe('') -> FAILED (empty string), returned []
3. searchProductSafe('<script>') -> FAILED (has < > chars), returned []
4. loginSafe('admin', 'admin123') -> PASSED validation, login successful
5. loginSafe('admin', 'ab') -> FAILED (password too short), returned null
6. loginSafe('ad min', 'pass123') -> FAILED (space in username), returned null

Commands I ran:
node secure.js

Why validation matters:
Parameterised queries protect the database from SQL injection, but input validation protects the whole application. For example, someone could still put HTML tags like <script> into the database which might cause XSS attacks later. Validation stops bad data before it ever gets stored. This is called "defense in depth" - multiple layers of security.
